### What is NetGameSim actually doing?

The main idea of NetGameSim is to simulate how a network changes over time.

1. **Generation:** It builds a random graph (e.g., 300 nodes connected in a specific way)
2. **Perturbation (The "Chaos" Step):** It randomly adds/removes nodes and edges to simulate a changing network
3. **Delta Calculation:** It calculates the _difference_ between the original graph and the new "perturbed" one.
4. **Output:** It saves these graphs to your disk (`.ngs` files for data, `.dot` files for pictures).

### Understanding how the project is Organized

The project is structured as a multi-module sbt project. This
means that it is split into three distinct "sub-projects" that
work together.

#### `NetGameSim/` (The top-level folder)

The top-level folder job is to tie the other modules together and run the program.

- `build.sbt:` Tells the compiler that this project is made of three parts (`root`, `NetModelGenerator`, `GenericSimUtilities`) and lists the external libraries (like `Guava`, `Logback`, `JGraphT`).

- `src/main/scala/Main.scala:` The entry point. It reads the arguments, sets up logging, calls the generator, and saves the output.

- `outputs/`: The folder where the program writes the generated files (.ngs, .dot, and eventually your .json)

#### `NetGameSim/GenericSimUtilities/` (Base Configuration)

- `src/main/scala/Randomizer` (The Dice Rollers)

	- **Files:** `SupplierOfRandomness.scala`, `UniformProbGenerator.scala`

	- **What they do:** These are the math tools that generate random numbers.

	- **Why you care:** You need to implement "Node PDFs" (probability distributions) and random "Edge Labels". You will likely use these classes (or extend them) to make your random decisions (e.g., "There is a 20% chance this node sends a heartbeat").

- `src/main/scala/Utilz` (The Plumbing)
	- **Files:** `ConfigReader.scala`, `CreateLogger.scala`, `NGSConstants.scala`
	- `ConfigReader`: This is the code that parses the `application.conf` file. You will need to understand this to add your new config sections (like `node_pdfs` or `edge_labels`).
	- `NGSConstants`: This likely defines the default output paths and filenames. If you want to change where graph.json gets saved by default, this is the place to check.

- `src/main/resources/` (The Default Settings)
    - **Files:** application.conf, logback.xml
	- **What they do:** This is the "Master" configuration file. It defines every possible setting available in the program and sets their default "fallback" values (e.g., NetModel.statesTotal = 100).
	- **Why you care:** This is your "Reference Manual." You typically do not edit this file directly. Instead, you look at it to find the exact parameter names you need (like `NGSimulator.seed`), and then you override those values in the "Active" configuration file located in `NetModelGenerator`.

#### `NetGameSim/NetModelGenerator/` (The Factory)

- `src/main/scala/NetGraphAlgebraDefs` (The Blueprint & Builder)

	- **Files:** `NetGraph.scala`, `NetGraphComponent.scala`, `NetModelAlgebra.scala`, `GraphPerturbationAlgebra.scala`
	- **What they do:** This is the core logic. `NetModelAlgebra` reads your config and assembles the graph. `NetGraph` is the actual object holding all the data (nodes, edges). `GraphPerturbationAlgebra` handles the "random changes" (perturbations).
	- **Why you care:** This is the most important folder.
		- `NetGraph.scala`: This is where the graph object lives (g). You will add your `persistJson()` function here to export the data.
		- `NetGraphComponent.scala`: This defines what a "Node" and an "Edge" actually are. You will likely modify this to add your "Message Types" and "PDFs" to the node definitions.

- `src/main/scala/NetModelAnalyzer` (The Simulator)

	- **Files:** `Analyzer.scala`, `RandomWalker.scala`, `CostRewardCalculator.scala`
	- **What they do:** These scripts run simulations on top of the generated graph (e.g., simulating a user "walking" from node to node).
	- **Why you care:** For Phase 1 (Generating the Graph), you can mostly ignore these. However, looking at `RandomWalker.scala` might give you ideas for how to traverse the graph when you write your C++ logic later.
